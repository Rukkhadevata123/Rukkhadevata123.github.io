Gentoo Secure Boot 配置记录：shim + GRUB + MOK

重要

这是一份按实际使用方式整理过的配置记录。

参考 package.use

本文使用：

  • shim
  • GRUB
  • MOK
  • gentoo-kernel-bin 和 gentoo-sources 分开处理

没有使用：

  • sbctl
  • systemd-boot
  • UKI

一、整体思路

这里实际上用了 两套信任链，但都挂在 MOK 下面。

第一套：信任 Gentoo Distribution Kernel

目的：

  • 让 gentoo-kernel-bin 可以直接启动

做法：

  • 提取 distribution kernel 自带的签名证书
  • 把这个证书导入 MOK

这样一来，gentoo-kernel-bin 自己已经签好的内核镜像，就能直接被 Secure Boot 接受。

第二套：信任自己的自定义内核和模块

目的：

  • 给 gentoo-sources 编出来的内核模块签名
  • 给 NVIDIA 模块签名
  • 给自己编出来的 vmlinuz 签名

做法：

  • 自己生成一把 key
  • 把这把 key 的证书导入 MOK
  • 用这把 key 给模块和 EFI / kernel 文件签名

这样 make install 后的自定义内核，加上自动签好的模块，就能在 Secure Boot 下正常使用。

二、启动链是什么

走的是这条链：

UEFI Secure Boot
  -> shimx64.efi
  -> grubx64.efi
  -> Linux kernel

也就是说：

  • UEFI 先信任 shim
  • shim 再去启动已签名的 GRUB
  • GRUB 再加载已经被信任的内核

对应 EFI 分区里的关键文件大概是：

/efi/EFI/gentoo/shimx64.efi
/efi/EFI/gentoo/mmx64.efi
/efi/EFI/gentoo/grubx64.efi
/efi/EFI/gentoo/grub.cfg

其中：

  • shimx64.efi：shim 入口
  • mmx64.efi：MokManager
  • grubx64.efi：已签名的 GRUB EFI 文件
  • grub.cfg：GRUB 配置文件

三、先把 shim 和 GRUB 摆好

最终保留的是 shim + grub 这一套，不是 systemd-boot。

安装相关包：

emerge --noreplace sys-boot/grub sys-boot/shim sys-boot/mokutil sys-boot/efibootmgr

然后把 shim 和已签名的 grub 放到 ESP：

cp /usr/share/shim/BOOTX64.EFI /efi/EFI/Gentoo/shimx64.efi
cp /usr/share/shim/mmx64.efi /efi/EFI/Gentoo/mmx64.efi
cp /usr/lib/grub/grub-x86_64.efi.signed /efi/EFI/Gentoo/grubx64.efi

GRUB 配置这里放在 ESP 上，而不是默认的 /boot/grub/grub.cfg：

echo 'GRUB_CFG=/efi/EFI/Gentoo/grub.cfg' > /etc/env.d/99grub
env-update && source /etc/profile
grub-mkconfig -o /efi/EFI/Gentoo/grub.cfg

最后创建 EFI 启动项：

efibootmgr --create --disk /dev/sda --part 1 --loader '\EFI\gentoo\shimx64.efi' --label 'GRUB via shim' --unicode

这里的磁盘和分区号按你自己的机器改。

四、第一把“被信任的 key”：Gentoo distribution kernel 的 key

这一部分是为了让 gentoo-kernel-bin 直接启动。

distribution kernel 本身已经签过名，但 Secure Boot 不会天然信任它；你还需要把对应证书导入 MOK。

导入命令：

openssl x509 -in /usr/src/linux/certs/signing_key.x509 -outform der -out dist_mok.der
mokutil --import dist_mok.der

导入后，重启进入 shim 的 MOK 界面完成确认。

然后可以检查：

mokutil --list-enrolled | less
sbverify --list /boot/vmlinuz-7.0.11-gentoo-dist

这条线的意思很明确：

  • gentoo-kernel-bin 内核镜像是 Gentoo 官方签的
  • 只负责把它的证书加入 MOK
  • 这样它就能直接被 GRUB 加载

五、第二把“自己用的 key”：自定义内核和模块

这一把 key 才是自己真正长期要维护的 key。

它负责：

  • gentoo-sources 自定义内核模块签名
  • NVIDIA 模块签名
  • 自己编译出来的 vmlinuz 签名
  • 需要手动处理的 EFI 文件签名

1. 生成 key

生成 key：

openssl req -new -noenc -utf8 -sha256 -x509 -outform PEM -out kernel_key.pem -keyout kernel_key.pem
chown root:root kernel_key.pem
chmod 400 kernel_key.pem

然后导出证书给 MOK：

openssl x509 -in kernel_key.pem -inform PEM -out kernel_key.der -outform DER
mokutil --import kernel_key.der

同样需要重启，在 MOK 界面里确认导入。

六、make.conf 里这些变量到底是干什么的

这里最关键的是这几个：

MODULES_SIGN_KEY="/home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem"
MODULES_SIGN_CERT="/home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem"
MODULES_SIGN_HASH="sha512"

SECUREBOOT_SIGN_KEY="/home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem"
SECUREBOOT_SIGN_CERT="/home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem"

要让这套设置真正生效，还需要相关包启用对应 USE flag：

  • modules-sign
  • secureboot

这里要分开理解。

MODULES_SIGN_*

这组变量的作用是：

  • 给内核模块签名
  • 包括 gentoo-sources 相关模块
  • 也包括 NVIDIA 这种外部模块

实际用途是：

  • 同一把 key 负责给自定义模块和 NVIDIA 模块签名

这也是为什么要把自己的 key 导入 MOK，不然模块签了也没法在 Secure Boot 下通过验证。

SECUREBOOT_SIGN_*

这组变量主要用于：

  • 给需要签名的 EFI 文件做签名

例如：

  • grub 本体
  • 其他通过 ebuild / install 流程可自动签名的 EFI 文件

但它不等于会自动帮所有自编译出来的 vmlinuz 做签名。

这也是后面还要手动 sbsign 内核镜像的原因。

七、自编译 gentoo-sources 内核时的处理方式

工作流大概是：

zcat /proc/config.gz > .config
make LLVM=1 LSMOD=/home/yoimiya/.config/modprobed.db localmodconfig
make LLVM=1 nconfig
make LLVM=1 modules_prepare
make LLVM=1 -j16
make modules_install
emerge @module-rebuild
make install

还需要在：

make LLVM=1 nconfig

里设置：

CONFIG_MODULE_SIG_KEY="/home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem"

原因是：

  • make modules_install 是内核构建系统自己的行为
  • 它和 Portage 本身没有直接关系
  • 所以内核模块签名这一步不能只依赖 make.conf 的环境变量

可以把它理解成两层：

  • make.conf 里的 MODULES_SIGN_*
  • 内核配置里的 CONFIG_MODULE_SIG_KEY

两边都配好，自编译内核安装出来的模块签名才更稳。

其中：

  • make modules_install：装模块
  • emerge @module-rebuild：重建外部模块
  • make install：安装内核

因为已经在 make.conf 里设置了 MODULES_SIGN_*，所以：

  • 自定义模块
  • NVIDIA 模块

这两类模块会走同一把 key。

也就是说，这条线的目标就是：

  • 模块自动签
  • 内核镜像手动签

八、为什么 vmlinuz 还要手动签

例如：

sbsign vmlinuz-7.0.12-gentoo-r1-custom1 \
  --cert /home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem \
  --key /home/yoimiya/dotfiles/gentoo-openrc-hyprland/kernel_key.pem \
  --output vmlinuz-7.0.12-gentoo-r1-custom1

这里的意思是没有走：

  • UKI 自动签名
  • systemd-stub
  • sbctl 管理整个链条

而是选择了最直接的方式：

  1. make install
  2. 去 /boot
  3. 对新生成的 vmlinuz-* 手动 sbsign
  4. sbverify --list 检查
  5. 重启

验证命令：

sbverify --list /boot/vmlinuz-7.0.11-gentoo-custom1

九、最终可以把整个流程总结成这样

路线 A：gentoo-kernel-bin

目标：

  • 直接信任 Gentoo 官方签名的 distribution kernel

做法：

openssl x509 -in /usr/src/linux/certs/signing_key.x509 -outform der -out dist_mok.der
mokutil --import dist_mok.der

结果：

  • gentoo-kernel-bin 可直接启动

路线 B：gentoo-sources

目标：

  • 用自己的 key 信任自定义内核和模块

做法：

openssl req -new -noenc -utf8 -sha256 -x509 -outform PEM -out kernel_key.pem -keyout kernel_key.pem
openssl x509 -in kernel_key.pem -inform PEM -out kernel_key.der -outform DER
mokutil --import kernel_key.der

在 make.conf 中设置：

MODULES_SIGN_KEY="/path/to/kernel_key.pem"
MODULES_SIGN_CERT="/path/to/kernel_key.pem"
MODULES_SIGN_HASH="sha512"

SECUREBOOT_SIGN_KEY="/path/to/kernel_key.pem"
SECUREBOOT_SIGN_CERT="/path/to/kernel_key.pem"

同时确保相关包启用了：

modules-sign secureboot

编译安装后再手动签内核镜像：

sbsign /boot/vmlinuz-你的内核 \
  --cert /path/to/kernel_key.pem \
  --key /path/to/kernel_key.pem \
  --output /boot/vmlinuz-你的内核

结果：

  • 自定义内核模块和 NVIDIA 模块走自己的 key
  • 自定义内核镜像也走自己的 key

十、平时实际要记住的只有几件事

1. 新装或切换 gentoo-kernel-bin

记住：

  • distribution kernel 自带签名
  • 只要对应证书已经导入 MOK，就能直接启动

2. 自己重新编译 gentoo-sources

记住：

  • 模块会走自己的签名 key
  • NVIDIA 模块也会走这把 key
  • vmlinuz 还要手动 sbsign

3. GRUB 相关

记住：

  • 启动入口是 shimx64.efi
  • grub.cfg 在 /efi/EFI/Gentoo/grub.cfg
  • 更新内核或 GRUB 后别忘了：
grub-mkconfig -o /efi/EFI/Gentoo/grub.cfg

十一、现在的理解

这套方案的优点是：

  • 很传统
  • 很 Gentoo
  • 每一层都知道是谁在签、谁在信任谁
  • 出问题时容易排查

缺点也很明显：

  • 分成两把 key / 两条信任思路，脑子里得分清
  • 自编内核更新后还要手动签一次
  • 没有 UKI 那种“一整个文件完事”的简洁感

不过对来说，这套方案的优先级不是“最优雅”，而是“自己能控制、能理解、坏了能修”。
